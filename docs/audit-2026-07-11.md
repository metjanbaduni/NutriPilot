## 2. Severity-ranked issues

### Critical

**C1. The backend integration is unfinished ‚Äî the profile feature cannot work against real AWS.**
`amplify/backend/backend-config.json` and `amplify-meta.json` register exactly one function (`nutripilotFunction`) and one API path (`/api` ‚Üí `nutripilotFunction`). `getProfile/` and `updateProfile/` contain only `src/` ‚Äî they were never registered via `amplify add function` and are not routed through API Gateway. Meanwhile `src/api/profile.js` calls `GET/POST /profile`, a path that does not exist in the deployed API. The 94 green tests never catch this because they mock the Amplify API client.
Note this is **not** unstarted backlog: tasks.md marks T023/T024 (the Lambdas) done, and it's telling the truth ‚Äî the tasks only asked for the handler source files. **No task for Amplify registration / API Gateway routing / `amplify push` verification exists anywhere in tasks.md**, so the checkbox model shows US2 complete while the endpoint doesn't exist. Root cause analysis in section 4.
*Fix:* register both Lambdas with Amplify, add `/profile` (GET/POST) API Gateway routes with the Cognito authorizer, `amplify push`, and add at least one manual/E2E verification against the real endpoint. This is the exact pattern the Dashboard screen must copy, so it blocks Dashboard directly.

**C2. The deployed API is unmodified Amplify Express boilerplate.**
`amplify/backend/function/nutripilotFunction/src/app.js` is the stock scaffold: it answers every `GET/POST/PUT/DELETE /api*` with `"get call succeed!"` and **echoes request bodies back**, with `Access-Control-Allow-Origin: *` and `Access-Control-Allow-Headers: *`. It's behind Cognito auth (`private` setting), which limits exposure, but it's dead code deployed to production infrastructure.
*Fix:* either implement it as the real router or (better) delete it once C1 replaces it with purpose-built Lambdas; tighten CORS to the app origin.

### Should fix

**S1. Error-message handling violates the constitution in three places.**
- `RegisterForm.jsx:44-50` ‚Äî `getRegistrationErrorMessage` returns raw `error.message` (leaks Amplify internals) and, worse, falls back to *"Email already registered. Please sign in."* for **any** error without a message ‚Äî a network failure tells the user their email is taken.
- `LoginForm.jsx:30-32` ‚Äî every sign-in failure becomes "Invalid email or password", including `UserNotConfirmedException` (an unconfirmed user is told their password is wrong, with no path to re-confirm) and network errors.
- `ProfileForm.jsx:375` ‚Äî displays raw `error?.message`, which via `client.js:37-43` passes through backend `response.data.message` and raw Amplify network errors.
*Fix:* map known error codes to specific friendly messages, use a generic fallback for everything else, never display raw `error.message`.

**S2. `calculateMacros` is duplicated and already drifting.** `src/utils/calculateMacros.js` (ESM) vs `amplify/backend/function/lib/calculateMacros.js` (CJS) ‚Äî the JSDoc has already diverged. A silent formula drift here means the UI and the stored targets disagree. *Fix:* one source of truth (the `lib/` copy) with a documented sync mechanism, or a test asserting both files' outputs match on a fixture grid.

**S3. The constitution's mechanical rules are not actually enforced.** `eslint.config.js` has `complexity: 10` and `max-len: 100` but **no `max-lines-per-function: 50` and no `max-depth: 3`** ‚Äî the two rules the constitution says are tool-enforced. (Spot-checking, current code passes both; nothing is enforcing that stays true.) *Fix:* add both rules; also delete the dead legacy `.eslintrc.js` that ESLint 9 ignores.

**S4. No CI exists.** Constitution says "CI will fail if tests don't pass / coverage < 80% / ESLint errors"; workflow-v2 Step 7 says "CI check: gh run watch". There is no `.github/workflows/` directory. *Fix:* one workflow running `npm run verify && npm run build` on push/PR.

**S5. `SessionContext.jsx:99-102` rethrows fatal auth errors with no ErrorBoundary anywhere** ‚Äî any unexpected `getCurrentUser` error white-screens the app. The comment says error boundaries come "in later tasks"; that debt should be paid before adding more authenticated screens.

**S6. Design tokens exist in two disconnected places, plus a contradictory Tailwind config.** The shipped UI styles are hand-written classes in `src/index.css` (only 6 `var(--‚Ä¶)` usages); the imported design system in `docs/design/tokens/` is not referenced by the build; `tailwind.config.cjs` defines an unused **light-theme** palette (`surface: '#F5F7FB'`, `ink: '#0F172A'`) that contradicts the dark-only app. Ad-hoc values are already leaking: `text-rose-200` in `ProfileForm.jsx:124`, hardcoded gradient hexes in `AuthShell.jsx:24-25`. In practice the app **is** consistently dark-mode-only ‚Äî but the token plumbing that would keep the Dashboard consistent isn't wired. *Fix:* import `docs/design/tokens/index.css` (or fold the tokens into `index.css`), map them into Tailwind's theme, delete the light palette.

**S7. Old-workflow tooling is still live context for agents.** `.codex/config.yml` references root-level `constitution.md`/`plan.md` that don't exist there; `AGENTS.md` and `.github/copilot-instructions.md` still describe the Codex/Copilot flow; `scripts/prompts/` and the `prompt:*` npm scripts are Codex-era. Exactly the "old instructions leaking into agent context" risk workflow-v2 warns about ‚Äî except it names a file (`00-MASTER-GUIDE.md`) that doesn't exist while missing the ones that do.

### Cosmetic

- `LoginForm`/`RegisterForm` duplicate `validateEmail`/`validatePassword`/`EMAIL_PATTERN` verbatim ‚Äî extract to a shared module.
- `<a href="/login">` / `<a href="/signup">` cause full page reloads; use router `<Link>` (`RegisterForm.jsx:302`, `LoginForm.jsx:138`).
- Enforcing the full password policy on **login** (`LoginForm.jsx:14-28`) blocks sign-in before Cognito ever sees it and reveals the policy; validate only non-empty at login.
- `useMemo(() => session, [session])` in `SessionContext.jsx:135` is a no-op; identical JSDoc duplicated on `ProfileFormContent` and `ProfileForm`; `ProfileFormLayout` hand-forwards 12 props that could be spread.
- Select options render raw enum values (`prefer_not_to_say`, `bulk`) as user-facing labels (`ProfileForm.jsx:252-256`).
- The nonbinary/prefer-not-to-say BMR offset of `-78` (`calculateMacros.js:46-47`) is an undocumented product decision ‚Äî deserves a "why" comment per the constitution's own rule.

**Verdict against constitution.md:** the implemented code itself would largely **pass** review ‚Äî functions ‚â§50 lines, complexity clean, naming and AAA test structure consistently followed, input validated at both UI and Lambda boundaries, no secrets committed (`aws-exports.js`, `team-provider-info.json`, `.env` all correctly untracked). What would fail review is the **system**: unwired backend (C1/C2), error-message rule violations (S1), unenforced lint rules (S3), and nonexistent CI (S4).

### 4.2 The structural gap: no infrastructure tasks are ever generated

C1 is not an execution failure ‚Äî the agent did exactly what T023/T024 asked. The failure is in **task generation**: the task list has no category for Amplify registration, API Gateway routes, `amplify push`, or against-real-AWS verification. "Deployment Notes" at the bottom says *"After each backend change, run `amplify push`"*, but a note without a checkbox, acceptance criteria, or DoD is invisible to an agent working task-by-task. Every backend "Acceptance" criterion in the file is satisfiable with mocks alone ‚Äî so a story can reach its checkpoint (*"Users persist accurate macro targets"*) without a single real HTTP request ever succeeding.

### 4.3 Recommended changes to how tasks.md is written/generated

1. **Every Lambda task gets a sibling wiring task (or wiring folded into its DoD).** A backend task is not done until: function registered in `backend-config.json`, route added with the Cognito authorizer, `amplify push` succeeds, and a smoke request against the dev endpoint returns the expected shape. Make that last item an explicit Acceptance line ‚Äî it's the one check mocks can't fake.
2. **Story checkpoints need an end-to-end criterion.** Each phase already defines an "Independent Test" (e.g., US2: *"verify DynamoDB receives PROFILE + TARGETS items"*) ‚Äî but nothing requires executing it. Add it to the User Story Quality Gate as a checkbox: "Independent Test executed against the dev environment and passed."
3. **Make checkbox updates part of shipping, not memory.** The Execution Defaults already say "Mark the task [x] when done" ‚Äî T026A/T044 show it gets skipped. Put "tasks.md updated" into the /ship skill's steps so it's mechanical.
4. **Use the Task Card Template for all tasks.** Phase 1‚Äì3 tasks are one-liners; post-review additions and US3 use the full card (Files/Notes/Tests/Acceptance/DoD). The full-card US3 tasks are exactly the right shape for agent consumption ‚Äî self-contained, verifiable, scoped. Generate all future tasks in that format.
5. **Prune stale cross-references.** "Dependencies & Execution Order" and "Parallel Execution Opportunities" still reference T032/T033, which the US3 note says were folded away; T007 references `src/index.js` but the file is `src/index.jsx`. Small, but agents take these files literally.
6. **DoD should say `npm run verify`** instead of listing `npm run lint`; `npm run format:check` separately (the script exists now ‚Äî T044).

6. **Archive Codex-era context:** move `.codex/`, `scripts/prompts/` + `prompt:*` scripts, `docs/ai-assisted-development-workflow.md`, and `.github/copilot-instructions.md` to `docs/archive/`; replace `AGENTS.md` content with a pointer to the new CLAUDE.md.
7. **Wire the design tokens** into the build and delete the light-theme palette from `tailwind.config.cjs` (S6) ‚Äî do this last; it can also be folded into the Dashboard task since that's the first consumer.

Items 1‚Äì5 are roughly one focused session. The ErrorBoundary (S5) and cosmetic items can ride along with the Dashboard task itself.
