# Repository Guidelines

## Project Structure & Module Organization
- `src/` contains app logic: `components/` for UI views, `api/` for Amplify calls, `utils/` for shared helpers, and `index.js` bootstraps.
- `public/` holds static assets served in development; `docs/` stores decision records that must match feature updates.
- `tests/` keeps Jest suites; mirror source folders to keep intent obvious.
- `amplify/` and `src/aws-exports.js` are generated by the Amplify CLI — edit through the CLI rather than manually.

## Build, Test, and Development Commands
- `npm install` syncs dependencies; run after pulling changes.
- `npm run dev` serves `public/` with `http-server` on port 8080 for local iteration.
- `npm test`, `npm run test:watch`, and `npm run test:coverage` execute Jest; ensure the 80% global thresholds stay green.
- `npm run lint` / `npm run lint:fix` enforce ESLint; `npm run format` / `npm run format:check` apply Prettier’s 2-space style.
- `./verify-setup.sh` confirms Node, npm, and Amplify prerequisites on new machines.

## Coding Style & Naming Conventions
- ESLint rules flag `no-unused-vars`, `no-console`, and `max-len` 100; resolve warnings before review.
- Prettier enforces semicolons, single quotes, and 2-space indentation—format staged files before committing.
- Components/hooks use PascalCase (`MealPlanner.jsx`), utilities stay camelCase (`calculateMacros.js`), and constants are UPPER_SNAKE_CASE.
- Store secrets in `.env` patterned on `.env.example`; never hard-code keys in tracked files.

## Testing Guidelines
- Place unit tests alongside sources as `*.test.js` or inside `tests/` mirroring the feature path.
- Use focused `describe` blocks and mock Amplify/network boundaries to keep suites fast.
- Maintain ≥80% coverage by running `npm run test:coverage`; add regression tests for every bug fix.

## Workflow Quality Standards
- A task is complete only when its listed tests pass and `npm run lint` + `npm run format:check` pass.
- Frontend tasks must include loading, empty, error, and populated states unless explicitly out of scope.
- If a task creates or changes a public hook, API helper, or component, add/update tests in the same task.
- Do not create follow-up test-only tasks unless explicitly approved by the PO.
- Match existing theme tokens/classes; if new styles are needed, add them to `src/index.css` or `src/styles/{feature}.css` and document.
- Each user story must have:
  - a manual test doc under `docs/manual_testing/`, and
  - an architecture/troubleshooting doc under `docs/architecture/`.

## Commit & Pull Request Guidelines
- Follow the existing conventional format (`feat|fix|chore|docs: short summary`) as seen in `git log`.
- Keep commits scoped to one change and include test updates or rationale.
- Pull requests need a concise summary, verification notes, linked issues, and UI screenshots when relevant.
- Ensure lint, format, and test commands pass before requesting review and mention backend changes that require `amplify push`.

## Environment & Security Tips
- Use Amplify CLI for infrastructure edits; document notable changes in `docs/`.
- Exclude `.env` and AWS credentials from commits; rotate secrets immediately if leaked.

## Review Acceptance (Single-Source Policy)
- Review feedback is **blocking** only if it violates spec.md, plan.md, .specify/memory/constitution.md, tasks.md scope, or required tests.
- Required tests must exist and pass when the task lists them; missing/failed required tests are blocking.
- Feedback outside task scope, stylistic preferences, or optional refactors are **non-blocking**.
- If a decision is unspecified, stop and ask before proceeding.
- When in doubt, resolve against the highest-priority docs: AGENTS.md → spec.md → plan.md → .specify/memory/constitution.md → tasks.md.
